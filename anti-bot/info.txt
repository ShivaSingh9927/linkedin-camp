This is the most critical part of the entire system. If the Chrome Extension is the bridge, the Campaign Execution Engine is the engine room. If this component is built poorly, LinkedIn's security algorithms will detect the bot activity instantly, and your users' accounts will be permanently banned.

To build this securely, we will use Playwright (it handles multiple tabs and contexts better than Puppeteer) combined with the Stealth Plugin to evade basic bot detection.

Here is the blueprint for your background worker and anti-bot emulation.

1. The Execution Architecture
You cannot run browser automation directly in your Express API requests (it will time out and crash your server). You must use a Queue system.

The Scheduler (Cron): Runs every few minutes. It queries the CampaignLead table for any row where nextActionDate <= now() and isCompleted == false.

The Queue (Redis + BullMQ or Trigger.dev): The Scheduler pushes these tasks (e.g., "Send Invite to Lead X") into a Redis queue.

The Worker (Node.js + Playwright): A separate Node.js process that listens to the queue. It picks up tasks one by one, launches a headless browser, executes the action, and updates the ActionLog and CampaignLead tables.

2. Anti-Bot & Human Emulation Rules
When feeding this to your AI, strictly enforce these human emulation rules for the Playwright script:

Rule 1: The Stealth Plugin. Always initialize Playwright with playwright-extra and the puppeteer-extra-plugin-stealth package to hide the fact that it's a headless browser.

Rule 2: Cookie Injection, Not Login. Never type the user's email and password into the LinkedIn login page. Always inject the li_at cookie directly into the browser context.

Rule 3: Residential Proxies. Every browser context must be routed through a proxy IP (proxyIp from the User table) that matches the user's geographic location.

Rule 4: Realistic Viewports and User-Agents. Set a standard desktop screen resolution (e.g., 1920x1080) and a modern User-Agent string.

Rule 5: The "Human" Delay. Humans do not click instantly. Add randomized delays (sleep(2000) to sleep(5000)) between every single action (page load, clicking a button, typing).

Rule 6: Human Typing. When typing a connection message, use Playwright's delay option to simulate human keystrokes (e.g., type('Hello', { delay: 150 })).

3. Core Worker Code (Playwright Blueprint)
Here is the exact boilerplate script you need to provide your AI to generate the worker logic for sending a connection request.

TypeScript
import { chromium } from 'playwright-extra';
import stealthPlugin from 'puppeteer-extra-plugin-stealth';
import { PrismaClient } from '@prisma/client';

// Add the stealth plugin to evade detection
chromium.use(stealthPlugin());
const prisma = new PrismaClient();

// Utility function for human-like randomized delays
const randomDelay = (min = 2000, max = 5000) => 
  new Promise(resolve => setTimeout(resolve, Math.floor(Math.random() * (max - min + 1) + min)));

export async function executeSendInvite(userId: string, leadId: string, customMessage: string) {
  // 1. Fetch user and lead data
  const user = await prisma.user.findUnique({ where: { id: userId } });
  const lead = await prisma.lead.findUnique({ where: { id: leadId } });

  if (!user?.linkedinCookie) throw new Error("No LinkedIn cookie found.");

  // 2. Launch Browser (using Proxy if available)
  const browser = await chromium.launch({
    headless: true, // Change to false for debugging locally
    args: ['--no-sandbox', '--disable-setuid-sandbox']
  });

  // 3. Create an isolated Context with the stolen cookie
  const context = await browser.newContext({
    viewport: { width: 1920, height: 1080 },
    userAgent: 'Mozilla/5.0 (Windows NT 10.0; Win64; x64) AppleWebKit/537.36 (KHTML, like Gecko) Chrome/120.0.0.0 Safari/537.36',
    // Inject proxy here in production: proxy: { server: user.proxyIp }
  });

  // Inject the li_at cookie
  await context.addCookies([{
    name: 'li_at',
    value: user.linkedinCookie,
    domain: '.linkedin.com',
    path: '/',
    httpOnly: true,
    secure: true
  }]);

  const page = await context.newPage();

  try {
    // 4. Navigate to the Lead's profile
    await page.goto(lead.linkedinUrl, { waitUntil: 'domcontentloaded' });
    await randomDelay(3000, 6000); // Simulate reading the page

    // 5. Scroll randomly to mimic a human reading the profile
    await page.evaluate(() => window.scrollBy(0, 500));
    await randomDelay();

    // 6. Click the "Connect" button
    // Note: LinkedIn frequently changes button selectors. Use text-based selectors where possible.
    const connectButton = page.getByRole('button', { name: 'Connect' });
    
    if (await connectButton.isVisible()) {
      await connectButton.click();
      await randomDelay();

      // 7. Add a custom message
      if (customMessage) {
        await page.getByRole('button', { name: 'Add a note' }).click();
        await randomDelay();
        
        // Type like a human
        await page.locator('textarea[name="message"]').type(customMessage, { delay: 100 });
        await randomDelay();
      }

      // 8. Send the invite
      await page.getByRole('button', { name: 'Send' }).click();
      await randomDelay();

      // 9. Log the success in the database
      await prisma.actionLog.create({
        data: { userId, leadId, actionType: 'INVITE', status: 'SUCCESS' }
      });
    } else {
      throw new Error("Connect button not found or already connected.");
    }

  } catch (error) {
    // Log the failure for analytics
    await prisma.actionLog.create({
      data: { userId, leadId, actionType: 'INVITE', status: 'FAILED', errorMessage: error.message }
    });
  } finally {
    // Always close the browser to free up server memory
    await context.close();
    await browser.close();
  }
}
4. Handling LinkedIn Layout Changes
One major advancement you can make over older tools is how you handle element selectors. LinkedIn changes its CSS classes and DOM structure constantly to break bots like this.

Avoid CSS classes: Never use selectors like page.click('.artdeco-button--primary'). They change weekly.

Use Accessibility Locators: Stick strictly to Playwright's getByRole, getByText, and getByLabel (as demonstrated in the code above). These rely on the text a human sees, which LinkedIn changes far less frequently.
